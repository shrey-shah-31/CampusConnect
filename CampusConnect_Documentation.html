1. Introduction

CampusConnect is a full-stack college placement portal that connects students, recruiting companies, and the placement office (admin). Students discover approved jobs, apply, track interviews and offers, and maintain profiles. Companies post roles (subject to moderation), review applicants, schedule interviews, and manage offers. Administrators approve accounts and jobs, monitor the pipeline, and export placement analytics.

The system runs on XAMPP (Apache, PHP, MySQL).

Technologies: Frontend — HTML5, CSS3, JavaScript, Tailwind CSS (CDN) on key pages; Backend — PHP with PDO, session-based authentication, JSON APIs; Database — MySQL with schema in database/schema.sql and optional automatic alignment in includes/db.php.

2. Objectives

  • Build a multi-role placement ecosystem (student, company, admin).
  • Implement secure registration and login with role-based access and API guards.
  • Support the placement workflow: jobs → apply → pipeline → interviews → offers → student accept/reject.
  • Use AJAX/Fetch for notifications and dynamic data where implemented.
  • Provide admin analytics (KPIs, trends, risk hints, resume scoring) and exportable reports (CSV/JSON).
  • Include resume builder with draft assistance, compensation display helpers, duplicate job prevention, and email hooks for interviews.

3. System Architecture

Three-tier architecture

  1. Presentation: PHP pages and assets under assets/css, assets/js; public entry index.php / home.php; role areas student/, company/, admin/.
  2. Application: Auth in auth/; APIs api/jobs.php, applications.php, interviews.php, offers.php, notifications.php; shared includes/helpers.php, session.php, db.php, mailer.php.
  3. Data: MySQL for users, profiles, companies, jobs, applications, interviews, offers, notifications.

Communication uses form posts, redirects, and JSON over HTTP for APIs.

4. Database Design

Configure the database name in includes/db.php; import database/schema.sql into MySQL. Runtime migrations may add columns (e.g. student_decision on offers, profile_photo_path on student_profiles).

Relationships: Company → jobs (1:M); student/job → applications (M:N table); application → interviews (1:M); application → offer (typically 1:1). Indexes are created in db.php for dashboard and API performance.

5. Functional Modules

  1. Authentication: auth/register.php, login.php, logout.php; login requires approved/active status.
  2. Student: dashboard, jobs with client-side search (filters.js), applications, profile, interviews, offers (accept/reject), resume_builder.php.
  3. Company: company/index.php — post jobs (pending approval), duplicate detection, view/update applications, schedule interviews, upload offers.
  4. Admin: admin/dashboard.php — queues, analytics, resume scores, risk heuristics; admin/review.php — approve/reject/remove; admin/reports.php — KPIs, trends, CSV/JSON export.
  5. Notifications: in-app messages for jobs, applications, status changes, interviews, offers; API + notifications.js.

6. Features (Detailed)

Students: secure login; one application per job; pipeline visibility; offer decisions; resume builder with downloadable text; compensation formatting helpers in PHP.

Companies: moderated job posting; application list scoped to own jobs; status workflow including auto-offer sync for hired/intern_offer_sent; interview scheduling with optional email (send_portal_mail); offer letter uploads.

Admins: user and job moderation; rich metrics (placement rate, active users, approval timing); fraud/risk scoring on pending items; CSV exports for placements and applications.

7. Security Implementation

  • Password hashing via password_hash / password_verify.
  • PDO prepared statements for SQL injection mitigation.
  • Role checks on pages (require_role) and APIs (require_api_role).
  • HTML output escaped with htmlspecialchars() where used in templates.
  • Production hardening: validate uploads (type/size), use SMTP (e.g. PHPMailer) instead of raw mail().

8. Advantages

  • End-to-end placement workflow in one application.
  • Clear separation of access for three roles.
  • Actionable admin dashboard and exportable reports.
  • Schema file plus optional DB bootstrap and incremental fixes.
  • Responsive UI using Tailwind on key screens.

9. Limitations

  • Email reliability depends on server mail configuration; SMTP integration recommended.
  • Resume “AI” assistance is template/rule-based, not an external LLM.
  • Web-only; no native mobile app.
  • No built-in video interview or calendar sync.
  • Admin user creation is manual (database).

10. Future Scope

  • PHPMailer/SMTP and notification templates.
  • Calendar (.ics) and interview reminders.
  • OAuth login; PWA or mobile app.
  • Skill–job matching and recommendations.
  • Audit logs and compliance-friendly exports.

11. Conclusion

CampusConnect demonstrates full-stack development with PHP/PDO, MySQL, and a modern responsive frontend for campus recruitment. It is suitable for academic projects and as a base for a real placement office with production configuration and security hardening.
